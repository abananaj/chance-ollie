# Compilation

1. Start with json/styles/blocks/ .jsonc files, which include styles for both core and custom blocks.
2. Compile all contents between "// вћЎпёЏ EXPORT:" and "// вќЊ END EXPORT" into styles/blocks.json. Strip comments, $schema, version, styles, blocks lines, unmatched curly braces and trailing commas. Wrap in "blocks": { ... }" and add opening and closing curly braces.
3. Combine blocks.json, with elements.jsonc, and global.jsonc into json/styles.json. Strip comments, $schema, version, styles, blocks lines, unmatched curly braces and trailing commas. Wrap in "styles": { ... }" and add opening and closing curly braces.
4. Combine all jsonc files in json/settings/ into json/settings.json. Strip comments, $schema, version, settings lines, unmatched curly braces and trailing commas. Wrap in "settings": { ... }" and add opening and closing curly braces.
5. Combine json/config/parts.json, patterns.json, and templates.json into json/config.json. Strip comments, $schema, version, config lines, unmatched curly braces and trailing commas. Wrap in "config": { ... }" and add opening and closing curly braces.
6. There should be three files in json/ settings.json, styles.json, and config.json. Combine these three files into theme.json (in that order).
7. Add $schema and version, then check again for comments, unmatched curly braces and trailing commas.
8. Compare to structure in [$schema](https://schemas.wp.org/trunk/theme.json) and confirm that all keys are in the correct place.

NOTE: If there are duplicate entries, include both in compiled files. Note in theme.json.md at the bottom if there were any duplicates or errors found. Include empty objects.

## Plan: Update theme.compile.js Pipeline

The current script reads pre-compiled `.json` intermediates that no longer match the source structure. The new pipeline reads `.jsonc` sources directly, handles comment-stripping, EXPORT markers, and produces intermediate files before the final `theme.json`.

### Key Changes

1. **Add a `.jsonc` parser** вЂ” strip `//` line comments and `/* */` block comments before `JSON.parse()`. One small utility function used throughout.

2. **New: Blocks stage** вЂ” scan all `styles/blocks/*.jsonc`, extract content between `// вћЎпёЏ EXPORT:` and `// вќЊ END EXPORT`. Merge extracted block objects, wrap in `"blocks": { ... }`, write `styles/blocks.json`.

3. **New: Styles stage** вЂ” parse `styles/global.jsonc` and `styles/elements.jsonc` (strip comments, extract their `styles` content), merge with the blocks output, wrap in `"styles": { ... }`, write `styles.json` to `json/`.

4. **Updated: Settings stage** вЂ” scan all `settings/*.jsonc`, strip comments, extract each file's `settings` content, deep-merge, wrap in `"settings": { ... }`, write `settings.json` to `json/`.

5. **Config stage** вЂ” read `config/parts.json`, `patterns.json`, `templates.json`, merge top-level keys, write `config.json` to `json/` (new вЂ” currently done in-memory only).

6. **Final assembly** вЂ” read the three intermediate files (`settings.json`, `styles.json`, `config.json`), combine in that order under the `$schema`/`version` header, write `theme.json`.

7. **Vite plugin wiring** вЂ” the `buildThemeJson()` export and standalone execution stay the same. Optionally add a `buildStart` hook alongside `writeBundle` so it also runs in dev/watch mode.

### Files to Modify
